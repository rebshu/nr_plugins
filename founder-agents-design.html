New Radicle · Founder Agent Architecture · March 2026

Simulating three minds. Design approach.

Architecture for building agents that think, decide, and communicate like Rebecca, Johny, and Kelly — individually and as a team. Four interaction modes, one portable core.

00 — The Core Idea

The personality kernel is the agent

Each founder agent is a Claude call with a rich system prompt — a "personality kernel" — that encodes their cognitive signature, decision-making pattern, communication style, and characteristic questions. The kernel is the hard part. Everything else (modes, interfaces, orchestration) is scaffolding around it.

The existing working style docs are unusually well-suited to this. They don't just describe what each person does — they describe how they process, which is exactly what a language model needs to simulate thinking patterns rather than just tone of voice.

Design principle

Simulate cognition, not just voice.

Most "write as this person" prompts produce surface-level tone mimicry. Your docs go deeper — they encode processing sequences (Rebecca's 6-step problem-solving order), decision entry points (Johny eliminates from outside-in, Kelly decomposes before deciding), and collaboration signals (what silence means, what pushback means). The agents should reproduce the thinking, not just the vocabulary.

01 — Architecture

Four layers, bottom up

The system is designed to be portable — the personality kernels work regardless of interface. Start with a Cowork skill, extract later if needed.

Layer 1
Foundation

Personality Kernels

Three system prompts, one per founder. Each encodes: cognitive signature, processing sequence, decision style, characteristic questions, communication patterns, collaboration signals, friction triggers, and relationship to the other two founders. These are the agents. Everything else calls them.

Layer 2
Context

Team Dynamics Layer

Injected when agents interact with each other. Encodes pair dynamics (Rebecca × Johny, Kelly × Johny, Rebecca × Kelly), team strengths, blind spots, and operating protocols. This layer is what makes the simulated meeting mode feel like your actual team, not three disconnected personalities.

Layer 3
Modes

Interaction Orchestration

Four modes of use (detailed below): solo consultation, panel review, simulated meeting, and voice drafting. Each mode wraps the personality kernels differently — different system prompts, different turn-taking logic, different output formats.

Layer 4
Interface

Invocation Surface

How you call the agents. The kernels are portable — a Cowork skill today, an API agent or MCP server tomorrow. The interface is the thinnest layer. Start simple.

02 — Kernel Structure

What goes inside each personality kernel

Each kernel is a system prompt with seven sections. The sections are ordered so the model reads cognitive patterns before communication style — processing drives output, not the reverse.

  • Identity & role
    Who you are and what this simulation is for. "You are simulating Rebecca, co-founder of New Radicle. Your role is to think through problems the way Rebecca actually thinks — not to imitate her tone, but to reproduce her cognitive process."
  • Cognitive signature
    How you process information. Entry mode (source-first / parallel mapping / decomposition). Processing style (sequential-deep / parallel-cross-pollination / 6-phase cycle). This is the most important section — it determines what the agent does when given a problem, not just what it says.
  • Decision pattern
    How you reach conclusions. Rebecca builds upward from source; position emerges from model. Johny eliminates via "what's missing?"; pressure-tests from outside in. Kelly leads with instinct, validates with research, commits cleanly. Each pattern produces visibly different decision outputs.
  • Characteristic questions
    The questions you always ask. Rebecca: "What's the source?" "Is this naming precise?" Johny: "What are we missing?" "How does this connect to X?" Kelly: "What are we actually deciding?" "What's the human question underneath this?" These are the most recognizable simulation markers.
  • Communication style
    How you express your thinking. Sentence structure, level of directness, how much scaffolding you provide, how you signal confidence vs. uncertainty, whether you lead with the conclusion or build toward it. Derived from the docs but also needs calibration against real messages/writing from each founder.
  • Friction & pushback triggers
    What you push back on and how. Rebecca pushes on imprecise naming as an architectural concern. Johny pushes on unaddressed gaps. Kelly pushes on confident answers that don't pass her smell test. These triggers make the simulation useful — an agent that only agrees is worthless.
  • Relationship to other founders
    How you see the other two. Pair dynamics, what you rely on them for, where you complement and where you clash. Only activated in multi-agent modes (panel review, simulated meeting). This is what produces realistic team interactions.
Calibration protocol

Each founder should read their own kernel and say "that sounds like me"

The working style docs are an extraordinary starting point — more detailed than most teams ever produce. But they're still descriptions of behavior, not examples of it. The kernels should be iteratively refined: generate a response, show it to the real founder, adjust. 2–3 rounds of calibration per founder will get the simulation from "plausible" to "uncanny."

03 — Interaction Modes

Four ways to use the agents

Each mode wraps the same personality kernels differently. You don't need all four on day one — start with solo consultation and panel review, add the others when you have the kernels calibrated.

Mode 01

Solo Consultation

Ask one founder for their take on something. "Rebecca, what do you think about this entity structure?" "Johny, what are we missing in this thesis?" The agent uses only its personality kernel. Simplest mode — and the best one for calibrating each kernel.

Mode 02

Panel Review

Run a proposal through all three founders sequentially. Each sees the previous founders' responses. Produces a structured output: Rebecca's take, Johny's take, Kelly's take, plus a synthesis of where they agree and diverge. This is the decision pressure-testing mode.

Mode 03

Simulated Meeting

Multi-turn conversation between the three agents. Each agent responds to the others with awareness of pair dynamics and team protocols. You set the agenda, then watch (or participate). Requires the team dynamics layer. Most complex mode — build last.

Mode 04

Voice Drafting

"Draft this investor update as Johny would write it." "How would Kelly frame this for a customer?" Uses the communication style section of the kernel heavily. Different from solo consultation — the output is a document, not a dialogue turn.

Build order

Solo → Panel → Voice → Meeting

Solo consultation is where you calibrate the kernels — it has the fastest feedback loop. Panel review is the highest-value mode for daily use (decision pressure-testing). Voice drafting is useful but lower-stakes. Simulated meeting is the most impressive and the hardest to get right — the agents need to genuinely respond to each other's arguments, not just monologue in character. Save it for after the kernels are solid.

04 — Panel Review Deep Dive

The highest-value mode, detailed

Panel review is likely the mode you'll use most. Here's how it works under the hood.

  • Step 1: You provide input
    A proposal, decision, draft document, strategy question, or anything you want pressure-tested. The input goes to all three agents along with a stakes flag: "This is a 70% decision" or "This is a 95% decision." The stakes flag changes how each agent responds — less research-oriented on 70%, more thorough on 95%.
  • Step 2: Rebecca responds first
    She goes to the foundation. Her response checks: Is the source material solid? Is the naming precise? Is the model built correctly from the bottom up? Her characteristic output: identifies what needs to be verified at source level, flags imprecise framing, and either confirms the architecture or says what's structurally wrong.
  • Step 3: Johny responds second
    He sees Rebecca's response. His entry point is different: What's missing? What doesn't connect? Where's the gap that invalidates? His characteristic output: surfaces the non-obvious connection or the missing dimension, names what the proposal hasn't addressed, and pressure-tests from outside in.
  • Step 4: Kelly responds third
    She sees both Rebecca and Johny. Her entry point: What are we actually deciding? Is the framing right? What's the human question underneath? Her characteristic output: reframes the decision if needed, names the sub-decisions, identifies what needs to happen before committing, and calls whether the team is over-processing a 70% call.
  • Step 5: Synthesis
    A fourth call (no personality kernel — just analytical) reads all three responses and produces: Points of agreement, Points of tension, Open questions, and a Recommended next step. This is the actionable output.
05 — Interface Recommendation

Where to start

The personality kernels are portable — they work in any interface that can make Claude API calls. The question is what to build first. Three options, with a clear recommendation.

Recommended
Option A

Cowork Skill

A skill you invoke right here. Fastest to build (hours, not days). Supports all four modes. Each founder's kernel lives as a markdown file; the skill reads them and orchestrates the calls.

Pros: Immediate to use, easy to iterate on kernels, no infrastructure, all team members can use it from their own Cowork sessions.

Cons: Tied to Cowork. Can't call it from other tools or programmatically.

Option B

MCP Server

An MCP tool server so any Claude client (Cowork, Claude Code, Claude Desktop) can call the founder agents as tools. More portable than a skill.

Pros: Works across Claude interfaces. Can be shared with the whole team. Clean tool semantics.

Cons: More setup (Python or TypeScript server). Needs to be running to use. Overkill for initial calibration.

Option C

API Agent (Python)

A standalone Python script using the Claude API. Maximum flexibility — can be integrated into any workflow, called from scripts, chained with other tools.

Pros: Fully programmable. Can integrate with Slack, Notion, email. No dependency on Cowork.

Cons: Requires API key management. Most engineering effort. Not the right place to start unless you need programmatic access now.

Recommendation

Start with a Cowork skill. Extract later if needed.

The critical path is getting the personality kernels right. A Cowork skill lets you iterate on the kernels in minutes — change a line in the kernel file, re-invoke the skill, see the difference immediately. Once the kernels are calibrated and the modes are working, extracting to an MCP server or API agent is straightforward — the kernels are just text files.

06 — Build Plan

What to build, in order

  • Phase 1
    Kernels
    Write three personality kernel files. One per founder. Use the working style docs as raw material but restructure into the seven-section kernel format. This is the real work. Expect 2,000–3,000 words per kernel. Then have each founder read theirs and say what's right and what's off. 2–3 calibration rounds.
  • Phase 2
    Solo mode
    Build the Cowork skill with solo consultation only. Invoke with something like "ask Rebecca about X." This gives you the fastest feedback loop for kernel calibration. Test each agent against real decisions you've already made — does the agent's response match what the real founder actually said?
  • Phase 3
    Panel mode
    Add the panel review orchestration. Sequential calls, each agent seeing previous responses, synthesis at the end. Test against a real proposal the team has already debated — does the panel surface the same tensions and agreements the real conversation did?
  • Phase 4
    Voice + Meeting
    Add voice drafting and simulated meeting modes. Voice drafting is straightforward once kernels are solid. Meeting simulation requires the team dynamics layer and turn-taking logic — more orchestration work, but the hardest part (the kernels) is already done.
  • Phase 5
    Extract
    If needed, port to MCP or API. The kernel files and orchestration logic are the same regardless of interface. This is a packaging step, not a redesign. Do it when you need programmatic access or want other tools to call the founder agents.
07 — What Makes This Hard

The real risks

Simulating a person's thinking is harder than simulating their voice. The risks are specific and worth naming.

Risk 01

The agents agree too easily

Language models default to helpfulness. If you ask "Rebecca, what do you think of this?" the model's instinct is to be supportive. The kernels need to explicitly encode pushback triggers — what each founder would challenge, not just what they'd affirm. The friction patterns in the working style docs are the most important source material for this. An agent that only agrees is useless.

Risk 02

Tone mimicry without cognitive fidelity

It's easy to get an agent that "sounds like" someone — uses their vocabulary, their sentence structure. It's harder to get one that thinks like them. The difference: a tone-only agent says something that sounds right. A cognition-faithful agent arrives at the same conclusion through the same process, even when the conclusion is surprising. The kernel structure prioritizes processing patterns over communication style for this reason.

Risk 03

The simulation becomes a crutch for real conversation

The agents are useful as a preview — "how might Johny respond to this before I bring it to him?" They're dangerous as a substitute — "I already know what Kelly thinks because the agent told me." The simulation will always be a lossy compression of the real person. Build a norm: use the agents to prepare for real conversations, not to replace them.

Risk 04

The meeting simulation collapses into three monologues

In a real team meeting, founders respond to each other's actual arguments, not just their general positions. The simulated meeting needs genuine multi-turn interaction — Johny challenging Rebecca's framing, Kelly reframing Johny's question, Rebecca going back to source in response to both. Without the team dynamics layer and careful turn-taking design, you get three parallel speeches, not a conversation.

Next step

Ready to build Phase 1?

The next move is writing the three personality kernels. The raw material is already in the working style docs — it needs to be restructured into the seven-section kernel format, sharpened with specific examples, and loaded with the right pushback triggers. Once the kernels exist, building the Cowork skill around them is fast. The kernels are the product. Everything else is delivery mechanism.
